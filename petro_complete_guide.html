Section 1 — Overview

What is PETRO?

Imagine seeing every instance of burning natural gas in oil fields worldwide in real-time, from satellites in space — knowing which company is responsible, how much CO₂ is being released, and the financial value of the wasted gas. That is PETRO.

PETRO is a dashboard and backend intelligence system that tracks global oil & gas flaring (the burning of natural gas during oil extraction) using NASA satellite data. It detects, tracks, and quantifies flaring events by downloading daily thermal data from NASA satellites, identifying which flares belong to which company, calculating emissions and gas value from heat signatures, alerting on unusual spikes, and visualising everything on a global interactive map.

What is "Flaring"?

When oil companies drill for oil, they also produce natural gas as a byproduct. Instead of capturing the gas for sale, reinjecting it, or storing it, companies often burn it off at a flare stack. This happens because capture equipment is expensive, gas market prices may be low, infrastructure is unavailable at remote sites, or regulations allow it. The burning releases massive amounts of CO₂ and wastes gas that could otherwise be sold.

Section 6 — Pipeline

Data Flow & Processing Pipeline

Fetch — NASA FIRMS API pulls ~8 regions × 7 days. Each detection: lat, lon, brightness, FRP, timestamp. Result: DataFrame ~10,000 rows.
Filter — Keep only industrial flares (type 2 = static land, type 3 = offshore). Brightness > 1600 K OR FRP > 1 MW. Removes agricultural fires and volcanoes. Result: ~5,000 rows.
Cluster — DBSCAN with Haversine metric (eps = 1 km). Groups nearby detections into flare "sites". Example: 50 pixels in a 1 km area → 1 flare site. Result: ~500 unique flare sites.
Attribute — Match each flare to a known well within 5 km. If no exact match, use basin probability weighting. Extracts company name, country, basin, confidence score.
Calculate emissions — FRP → Gas Volume (Elvidge 2016 polynomial). Gas Volume → CO₂ (IPCC AR6 factor 2.86 kg/kg). Gas Volume → USD value (energy content × market price).
Detect anomalies — For each cluster, compute 7-day rolling baseline + σ. Flag as anomaly if today > baseline + 2σ. Assign severity: ALERT / HIGH / CRITICAL.
Cache — Store result in memory for 1 hour (TTL). Subsequent requests return cached copy instantly.
Serve — REST API returns GeoJSON FeatureCollection. Frontend fetches and renders on the global map.

Section 9 — Atmospheric

Wind Vector & Atmospheric Plume Simulator

PETRO fetches real-time 10-metre wind vectors from Open-Meteo (free, no API key) and models how emissions disperse downwind of each detected flare using Gaussian plume dispersion mathematics.

How plume modelling works

Get current wind u10 (eastward) and v10 (northward) components from Open-Meteo Forecast API
Model Gaussian plume dispersion: source = flare location + FRP magnitude; stability class varies by time of day and surface type
Calculate downwind concentration gradient at increasing distances
Render as semi-transparent coloured contour layer on the map. Blue = low concentration, red = high.

Section 12 — Science & Algorithms

Advanced Concepts

DBSCAN Clustering Algorithm

Groups nearby satellite pixels into distinct flare sites. Runs on all detection coordinates with eps = 1 km and min_samples = 1 using the Haversine metric to respect Earth's curvature.

Key parameters

eps = 1 km — distance threshold (smaller = more clusters)  ·  min_samples = 1 — even single detections get a cluster ID  ·  Haversine metric — accurate geographic distance on a sphere  ·  Complexity: O(n log n) via ball-tree, <1 second for 10,000 points

Elvidge 2016 Emission Model

Peer-reviewed formula (Remote Sensing journal, 2016) converting satellite heat signature to natural gas volume. Based on measurements from oil fields in Kazakhstan, validated across multiple regions.

log₁₀(V_gas) = 1.40 + 1.55 × log₁₀(FRP_MW)
V_gas = 10^(1.40 + 1.55 × log₁₀(FRP))

Example: FRP = 15.3 MW → log₁₀(15.3) = 1.185 → exponent = 3.237 → V_gas ≈ 1,726 scm/day

Limitations: Assumes complete combustion, gas composition varies by region, atmospheric conditions affect observation, only valid for industrial flares.

CO₂ Emission Calculation

Gas volume → mass: mass_kg = volume_scm × 0.8 kg/m³ (standard natural gas density)
Mass → CO₂: CO₂_kg = gas_mass_kg × 2.86 kg CO₂/kg gas (IPCC AR6 factor). Why 2.86? Natural gas ≈ 75% methane (CH₄). Burning: CH₄ + 2O₂ → CO₂ + 2H₂O. Molecular weights give ~2.86 kg CO₂ per kg gas.
Regional correction: Middle East × 1.00, Niger Delta × 1.05, Permian Basin × 0.92, North Sea × 0.88, West Siberia × 1.02, Marcellus × 0.94
Methane uplift: CO₂_eq = CO₂_base × 1.003 (adds ~0.3% for fugitive methane leaks at 28× warming potential)
Validation check

Literature: ~200–300 kg CO₂ per MW of FRP. Our model: 3,956 kg ÷ 15.3 MW = 258 kg CO₂/MW ✓ (within expected range)

Gas-to-Market-Value Calculation

Energy content: energy_MJ = volume_scm × 38 MJ/scm (Gross Calorific Value per ISO 6976)
Convert to MMBtu: energy_MMBtu = energy_MJ ÷ 1055.06 (exact SI conversion)
Market value: value_USD = energy_MMBtu × $3.50/MMBtu (Henry Hub benchmark, typical 2026 price. Configurable in emission_factors.py)

Anomaly Detection — The 2σ Rule

Every flare has a 7-day rolling baseline of "normal" activity. A Z-score measures how far today deviates in units of historical standard deviation.

z_score = (FRP_today − FRP_baseline) ÷ FRP_std_dev

If today = 21 MW, baseline = 12.4 MW, σ = 1.2 MW → z = 7.17. Probability of random occurrence: <0.001%. The 2σ threshold means the anomaly will fire for events that naturally occur less than ~2% of the time.

Sigma levelSeverityNatural occurrence rate
> 3σCRITICAL<0.1%
2.5σ – 3σHIGH<0.6%
2σ – 2.5σALERT~2–5%
< 2σNormal—

Attribution & Confidence Scoring

Two-stage pipeline matching satellite detections to company wells:

Stage 1

Deterministic matching (exact)

If satellite coordinate is within 5 km of a known well → match confirmed. Confidence 90–95%. Method: "Exact well match (5km)".

Stage 2

Probabilistic matching (fallback)

Identify basin from bounding box. Weight by each company's production share in that basin. Confidence 50–80%. Method: "Basin weighted probability".

Confidence rangeReliabilityRecommended use
80–100%Very highCan be reported as fact
50–80%ModerateCross-reference before publishing
< 50%LowTreat as indicative only

Section 14 — Troubleshooting

Troubleshooting

Backend: can't reach port 8000
Ensure you are in the backend/ directory and run uvicorn main:app --reload --port 8000. If port 8000 is occupied, use --port 8001 and update the frontend API base URL accordingly.
FIRMS_MAP_KEY not found
Check that .env exists in the project root and is formatted without spaces: FIRMS_MAP_KEY=your_key_here. Run cat .env to verify.
No module named 'fastapi'
Run pip install -r requirements.txt inside the backend/ directory. For conda: conda install -c conda-forge fastapi uvicorn pandas numpy scikit-learn
Request timeout on NASA FIRMS data
NASA API can take 60+ seconds under load. Check your internet connection and retry in a few minutes. This is not a local error.
All companies showing "Unknown"
Excel well files are missing from the project root. Verify presence of: Saudi Aramco.xlsx, wells_india.xlsx, wells_iran.xlsx, wells_iraq.xlsx, wells_usa.xlsx
Mapbox token invalid / map shows grey
Obtain a valid token from account.mapbox.com. Set VITE_MAPBOX_TOKEN=your_token in .env and restart the frontend with npm run dev.
No anomalies detected
This is normal if flaring activity is stable. Anomalies naturally occur in only 2–5% of observations. Try a different date range or wait for actual spikes.
Dashboard slow / sluggish
Reduce date range to 7 days. Ensure internet speed > 1 Mbps. Clear browser cache (Ctrl+Shift+Delete). Chrome is the fastest browser for Mapbox rendering.
